# Orchestration core — shared dispatch / collect / synthesize / archive routines

This file is the **single source of truth** for the mechanics of a counselors run. It is read by **both** executors:

- the **skill itself**, acting as the *inline coordinator* for `run` mode (executed in your current Claude Code session), and
- the **detached loop coordinator** agent, for `loop` mode (a separate Solo-managed process).

Each caller supplies the parameters below and wraps these routines with its own control flow:

- **run** = one wave, raw prompt, no rounds (see `SKILL.md`).
- **loop** = discovery + prompt-writing, then one wave per round, convergence, timeout (see `coordinator-loop-prompt.md`).

Identity, status lifecycle, cancellation polling, round control, and done-signaling belong to the **caller**, not here. These routines start once identity and parameters are established.

## Shared conventions

- **Scratchpad gotcha**: `scratchpad_write` takes a `name` and **returns a `scratchpad_id`**. Every *other* op — `scratchpad_read`, `scratchpad_tail`, `scratchpad_archive` — is keyed by `scratchpad_id`, NOT by name. Enumerate a run's *visible* pads with `scratchpad_list(tags=["{run_id}"])` (returns ids + names; archived pads excluded). **Record the `scratchpad_id` of every pad you write** — you need ids to append, read back, and archive.
- **Worker-pad naming**: `counselors.{run_id}.worker.{round_seg}{agent}-{index}`, where `round_seg` is `""` (run) or `r{N}.` (loop round N) — yielding `…worker.{agent}-{index}` or `…worker.r{N}.{agent}-{index}`. Tags: `["counselor","{run_id}","worker","{agent}"]`, plus `"round-{N}"` in loop.
- **Verify-pad naming**: `counselors.{run_id}.verify.{verifier_agent}-{target_index}` — one per verifier, holding its verdicts on the worker it was assigned to check (`target_index` disambiguates, since each worker is verified once). Tags: `["counselor","{run_id}","verify","{verifier_agent}"]`. Written only when verification runs (see VERIFY-WAVE); archived with the other working pads on a clean `done`.
- **Findings index**: every worker ends its output with a mandatory machine-readable `## Findings index` (one pipe-delimited line per finding: `F<n> | severity | confidence | file:symbol | claim`, or a single `(none)`). Parse it from the fence-extracted text to deduplicate across the panel, count new findings per round (convergence), and target verification by stable id. The rich prose above the index is for the human reader.
- **Fence extraction**: pull the text between `===COUNSELOR-OUTPUT-BEGIN===` and `===COUNSELOR-OUTPUT-END===`. If the markers are missing, take the last ~3000 chars and prefix `[no fence markers detected — raw tail]`.
- **Read-only spawn**: under `read_only_policy == strict`, pass an agent-specific tool allowlist via `extra_args` at spawn (Claude Code: `["--allowedTools","Read,Glob,Grep,WebFetch,WebSearch"]`). The `--allowedTools` flag is Claude-CLI-specific, so gate it on the agent's `tool_type` from `list_agent_tools` (Solo reports `tool_type`: `claude`, `gemini`, `codex`, `copilot`, `generic`): apply the allowlist only when `tool_type == "claude"`. For any other `tool_type` — **no `--allowedTools` flag wired in** — do **not** refuse it and do **not** ask the user; automatically **downgrade just that agent to best-effort** (the worker preamble's in-prompt READ-ONLY block, which binds every backend) and note the per-agent downgrade in progress. Model diversity is the whole point of the panel, so keep every agent in the run: `strict` means "hard sandbox where supported, in-prompt enforcement everywhere else," never a silently shrunk panel.
- **Project context files**: backends auto-load only their *own* context file — Claude Code reads `CLAUDE.md`, codex and antigravity read `AGENTS.md`, gemini reads `GEMINI.md`; copilot and grok (Grok Build) read `CLAUDE.md` and `AGENTS.md` natively — so a repo whose conventions live only in `CLAUDE.md` is invisible to gemini/codex/antigravity workers, and they re-derive it from scratch. The skill closes the gap in-prompt, uniformly across backends and both modes: the worker and verifier preambles direct every agent to read whichever of `CLAUDE.md` / `AGENTS.md` / `GEMINI.md` / `.github/copilot-instructions.md` exist at the repo root before analyzing, and BUILD-EXECUTION-PROMPT's discovery (when `enhance == on`) reads those files itself and names the ones that exist in the execution prompt so workers go straight to them. Nothing for the caller to wire per-agent — it rides in the preambles and the prompt.
- **Timekeeping**: Solo has **no read-the-clock tool**, and timer waits are **relative**, so a run carries a relative `duration_ms` budget — the skill derives it from `--duration` by arithmetic (`30m` → `1_800_000`), never from a wall clock — instead of an absolute deadline. Express every wait as a relative `*_ms` value. The detached **loop** coordinator is a Solo agent, so it enforces the total budget with a one-shot Solo timer (`timer_set(delay_ms=duration_ms, …)` set once at start) and waits on workers with idle timers — never a sleep loop or `now`-arithmetic. The inline **run** coordinator is an *external* session (timer-body delivery to it isn't *guaranteed* — though it often arrives in practice), so it schedules each idle-timer arm as the primary wake but doesn't depend on the body: the `already_idle` harvest and a short sleep-poll grace cover the cases where it doesn't show up, **re-applied on every drain iteration** (see *Idle-drain pattern* below). It passes `duration_ms` as the wave's `max_wait_ms` ceiling and bounds the whole drain with a one-shot wave-deadline timer it polls (clock-free) rather than summing elapsed time.
- **Idle-drain pattern** (how every wave is collected): spawn and send in **batches** — all `spawn_agent` calls in one step, then all `send_input` calls in one step; never spawn-wait-spawn. Then harvest with **`timer_fire_when_idle_any`**, not `_all`: wake on the **first** worker to go idle, record+`close_process` it, and re-arm `_any` on the rest — a fast worker is never held hostage to a slow peer (the old wait-for-all barrier discarded every fast worker's lead). Each iteration, in order: (1) check the wave-deadline / cancel cutoff; (2) read the arm response's `already_idle` and process those finishers **first** — `_any` *ignores processes already idle when it is armed*, so skipping this step hangs the drain whenever the last workers go idle during your processing; (3) only if some are still busy (`waiting_on` non-empty) do you wait for the `WORKER-IDLE` body (Solo-agent caller) or sleep-poll-grace re-check (external session), then re-scan. The pattern is identical for workers and verifiers; the loop coordinator additionally passes an `on_finish` hook that re-dispatches a finisher for its next round (see COLLECT-WAVE).

## Routine: BUILD-EXECUTION-PROMPT

Turns the user's task into the (usually enriched) prompt that every worker receives. Both modes call this once before dispatching — run inline, loop in the coordinator. The caller should **prewarm the panel via SPAWN-WORKERS just before calling this**, so the workers' CLI boot overlaps the discovery phase below instead of being paid afterward; this routine does not touch those prewarmed processes.

**Inputs**: `user_prompt` (text or path — the caller has **already folded in any `--context` file contents**, so treat it as the complete task input; context handling is the caller's job, not this routine's), `prompt_source` (`inline`|`file`), `enhance` (`on`|`off`), `preset_path` (or empty), `prompt_writing_text`, `execution_boilerplate_text`, `progress_id`, `duration_ms`.

1. If `enhance == on`: follow `prompt_writing_text`. Run the **discovery** phase — explore the scoped repo with Read/Grep/Glob, guided by the preset's *Discovery focus* when `preset_path` is set; **cap discovery at roughly a third of `duration_ms` (half at the very most)** — the worker wave shares the same budget, so an over-long discovery starves it. Then the **prompt-writing** phase to produce a sharp, self-contained `base_prompt` (a longer, repo-grounded prompt — not the raw one-liner). If you cannot read the repo, fall back to `base_prompt = user_prompt` and note it in progress.
2. Else (`enhance == off`): `base_prompt = user_prompt` (resolve from file if it is a path), verbatim.
3. `execution_prompt = base_prompt + "\n\n" + execution_boilerplate_text`.
4. `scratchpad_write(name="counselors.{run_id}.prompt", content=execution_prompt, tags=["counselor","{run_id}","prompt"])` → **record `prompt_id`**. Append `[execution-prompt-built enhance={enhance} preset={name|none} words={wc}]` to progress.

**Returns**: `execution_prompt`, `prompt_id`. (The `prompt_id` is archived with the other working pads on a clean `done`.)

## Routine: SPAWN-WORKERS  *(prewarm — boots the CLIs so cold-start overlaps prompt-building)*

`spawn_agent` returns immediately, but the underlying agent CLI still takes a few seconds to boot to its prompt. Spawning the panel **as early as possible — right before BUILD-EXECUTION-PROMPT** — lets those boots run *under* the discovery phase that would otherwise block the workers, so by the time the execution prompt exists every worker is already booted, idle, and waiting. The dispatch send then lands instantly instead of after a cold start. Both modes prewarm round 1 / the single run wave this way; loop pipelining re-spawns later rounds on demand (their boot overlaps the peers still running in the drain, so they need no separate prewarm).

**Inputs**: `dispatch` (list of `{agent, agent_tool_id, index}`), `read_only_policy`, `name_suffix` (`""` for run, `-r{N}` for loop round N).

1. **Batch every spawn in one step** — issue all `spawn_agent` calls together so the boots run concurrently; do **not** spawn one, wait, then spawn the next. For each entry: `spawn_agent(agent_tool_id={entry.agent_tool_id}, name="counselor-{agent}-{index}{name_suffix}", include_agent_instructions=true)`, adding the `strict` `extra_args` allowlist when `read_only_policy == strict` **and** this agent's `tool_type == "claude"` (otherwise the per-agent best-effort downgrade — see *Read-only spawn*; never skip an agent or pause to ask). The policy and `tool_type` are both known before discovery, so the allowlist is applied here at prewarm.
2. Capture per entry: `worker_pid` and the response's optional `agent_instructions` (both returned **immediately** — keep `agent_instructions` for the dispatch turn).
3. **Then move straight on to the next step (BUILD-EXECUTION-PROMPT). Do NOT `get_process_status`, poll, or otherwise wait for the workers to finish booting.** `spawn_agent` already returned everything you need; the CLIs boot in the background as independent OS processes *while* you build the prompt — that background boot is the whole point. You reconcile any spawn/boot failure later at send time (DISPATCH-WAVE's liveness check), where a dead worker is simply dropped.
4. A failed spawn is appended to progress and **skipped**, not fatal.

**Returns**: `spawned` = `[{agent, index, worker_pid, agent_instructions}]` — each booting/booted, idle, holding no task yet.

Prewarm pays off most when `enhance == on` (there is a discovery phase to hide the boot under). With `enhance == off` the prompt is ready immediately, so SPAWN-WORKERS and DISPATCH-WAVE run back-to-back — still batched, just with nothing to overlap. **If the run is aborted between prewarm and dispatch, `close_process` the prewarmed workers** so no idle CLI is left behind.

## Routine: DISPATCH-WAVE

Hands each **already-spawned** worker (from SPAWN-WORKERS) its task, with all sends issued in one batch.

**Inputs**: `spawned` (from SPAWN-WORKERS — `[{agent, index, worker_pid, agent_instructions}]`), `execution_prompt`, `prior_round_context_by_index` (map from a worker's `index` to **that worker's own** prior-round block; empty/absent ⇒ no prior context — run mode, or loop round 1), `worker_preamble_text`.

1. For each spawned worker, build the **composite first turn** by substituting into `worker_preamble_text`:
   - `{{EXECUTION_PROMPT}}` ← `execution_prompt` — **the same text for every worker in the wave**.
   - `{{PRIOR_ROUND_CONTEXT}}` ← `prior_round_context_by_index[entry.index]` (empty string if absent). **Each worker gets only its OWN prior-round findings — never another agent's. No cross-model pollution: round N of `claude-0` carries only round N-1 of `claude-0`.**
   Then prepend the worker's `agent_instructions` (then a blank line) if non-empty, so it gets its own bootstrap before our preamble; otherwise the composite is just the substituted preamble.
2. **Batch every send in one step**: `send_input(process_id=worker_pid, input=composite, submit=true, wait_ms=1500)` for all workers, issued together so they dispatch concurrently — the `wait_ms` is a boot-cushion paid **once in parallel**, never summed across workers (and pure slack when the workers were prewarmed and already idle).
3. **One batched liveness check**: `get_process_status` each. On a worker that didn't accept input, append an error line to progress and **continue** — the COLLECT-WAVE drain's one-shot nudge recovers a worker that sits idle with no output, and one bad send does not abort the wave.
4. Track `{agent, index, worker_pid}`.

**Returns**: the dispatched workers `[{agent, index, worker_pid}]`.

**Single-worker dispatch** (loop pipelining): to launch one worker for its next round, call SPAWN-WORKERS with a one-entry `dispatch` (`name_suffix = -r{N}`), then run steps 1–4 for just that worker, passing `prior_round_context_by_index = {that index: its own condensed prior-round block}`. Its boot overlaps the peers still running in the drain, so it needs no separate prewarm.

## Routine: COLLECT-WAVE  *(per-finisher idle-drain)*

Harvests workers **as each one goes idle**, never blocking a finished worker on a slower peer. Implements the *Idle-drain pattern* (conventions) with `timer_fire_when_idle_any` — the old wait-for-all `timer_fire_when_idle_all` barrier left every fast worker's output unscraped and its slot occupied until the slowest finished.

**Inputs**: `workers` (from DISPATCH-WAVE), `round_seg` (`""` or `r{N}.`), `extra_tags` (`[]` for run, `["round-{N}"]` for a single-round loop wave), `progress_id`, `wave_budget_ms` (relative ceiling for the whole drain). When a worker entry carries its own `round` (the pipelined loop drain, where concurrent workers may be on different rounds), that per-worker round wins for *its* pad name and round tag; the wave-level `round_seg`/`extra_tags` are the fallback for callers whose workers all share one round (run mode, verify). **Optional**: `deadline_timer_id` (an already-armed authoritative cutoff to poll instead of arming a new one — the loop coordinator passes its run-deadline timer so the pipelined drain is bounded by the run budget; absent ⇒ this routine arms its own wave-scoped cutoff from `wave_budget_ms`); `on_finish(worker, extracted)` (pipelining hook — see step 4); `cancel_check` (a `kv_get` key the caller wants polled each iteration).

**Cutoff (clock-free).** Solo has no clock, so the drain is bounded by a one-shot timer, never arithmetic. If `deadline_timer_id` was passed, poll it; otherwise `timer_set(delay_ms=wave_budget_ms, body="WAVE-DEADLINE {run_id}")` → `wave_deadline_id` and poll that. The cutoff has **fired** when its body arrives **or** `timer_list` no longer shows the id.

**Drain loop** — `remaining =` the set of worker entries still running (you arm `_any` on their `worker_pid`s); repeat until `remaining` is empty:

1. **Cutoff / cancel check (first, every iteration).** If the cutoff has fired → budget spent: for each `w` in `remaining`, best-effort `get_process_output(w.worker_pid)` + record the raw tail (status `timeout`) + `close_process(w.worker_pid)`; clear `remaining`; stop the drain. If `cancel_check` is set → stop now, return collected-so-far flagged `cancelled`, and dispatch nothing further (the caller handles cancellation).
2. **Arm `_any`**: `resp = timer_fire_when_idle_any(processes=[w.worker_pid for w in remaining], max_wait_ms=wave_budget_ms, body="WORKER-IDLE")`.
3. **Harvest `already_idle` FIRST (the `_any` gotcha):** `_any` *ignores processes already idle when armed*, so treat `resp.already_idle` as finishers to process **right now**. If `already_idle` is empty and `resp.waiting_on` is non-empty, a timer is pending → wait for the next wake: a Solo-agent caller (the loop coordinator) ends its turn and resumes on the `WORKER-IDLE` body (or the `max_wait_ms` / cutoff fire); an external-session caller (run mode) can't rely on body delivery, so it sleep-poll-grace re-checks `get_process_status` instead (see *Timekeeping*). On resume, determine the idle set by scanning `remaining` with `get_process_status`, and re-check the cutoff (step 1). *(Skipping the `already_idle` harvest hangs the drain when the last workers all idle during your processing — `_any` won't re-count them.)*
4. **Process each finisher `w`** (a worker entry in the idle set): `get_process_status(w.worker_pid)` (clean idle vs crash/timeout); `get_process_output(w.worker_pid, lines=4000)`; **fence-extract** (see conventions). If `w` is idle with **no closing fence**, hasn't been nudged, and the cutoff isn't imminent → `send_input(w.worker_pid, "You went idle without emitting your analysis. Proceed with your best read-only analysis now and wrap your final answer in ===COUNSELOR-OUTPUT-BEGIN=== / ===COUNSELOR-OUTPUT-END===.", submit=true)`, mark `w` **nudged**, and **leave it in `remaining`** (re-checked next iteration). Otherwise (fence present, or already nudged → fall back to the raw tail prefixed `[no fence markers detected — raw tail]`): write the worker pad — `scratchpad_write(name="counselors.{run_id}.worker.{seg}{agent}-{index}", content=extracted, tags=["counselor","{run_id}","worker","{agent}", ...tags])`, where `{seg}` and the round tag come from `w`'s **own** `round` when it has one (`r{w.round}.` and `"round-{w.round}"` — concurrent pipelined workers may be on different rounds) and otherwise from the wave-level `round_seg`/`extra_tags` → **record the returned `scratchpad_id`** in your working-pads list; keep the extracted text **in memory** (synthesis, and in loop the next round's prior-context); append `[{seg}{agent}-{index} done | status={status} | words={wc} | nudged={yes|no}]` to progress (you hold `progress_id`); `close_process(w.worker_pid)`; **remove `w` from `remaining`**. Then, if `on_finish` was supplied, call `on_finish(w, extracted)`; if it returns a freshly-dispatched worker entry (loop re-dispatch for the next round, carrying its incremented `round` and new `worker_pid`), **add that entry to `remaining`** so the drain keeps going.

**Cleanup**: if this routine armed its own `wave_deadline_id`, `timer_cancel` it (best-effort) once `remaining` is empty.

**Returns**: collected outputs (in memory, per worker) + the recorded worker `scratchpad_id`s + per-worker final status. With an `on_finish` re-dispatch hook the drain spans all rounds, so the returned pads/outputs cover every round each worker completed.

## Routine: VERIFY-WAVE  *(only when `verify_mode != off`)*

Adversarial cross-check of the panel's findings before synthesis. Each worker's findings are handed to **one independent peer — a different model where possible** — that re-reads the cited code and rules each finding `confirmed` / `refuted` / `uncertain`. This is the skill's defense against self-preferential bias (the synthesizer is itself a model, so an unverified finding from a same-family worker can sail into the report) and against confident-but-wrong findings (a plausible claim that the code doesn't actually support). It runs **once per run, after all finding-collection is done** — in run mode after the single COLLECT-WAVE; in loop mode after the final round — and before SYNTHESIZE. Verifiers are **fresh spawns** (the workers were already `close_process`ed) and are **read-only**, exactly like workers.

**Inputs**: `findings_by_worker` (map: worker `{agent, index}` → that worker's collected output text incl. its findings index — in loop, the worker's findings consolidated across all its rounds; you already hold these in memory), `panel` (the dispatch list `[{agent, agent_tool_id, index}]`, for resolving verifier `agent_tool_id`/`tool_type`), `verify_mode` (`cross`), `read_only_policy`, `verifier_preamble_text`, `progress_id`, `wave_budget_ms` (relative ceiling for this wave).

1. **Eligibility.** Let `producers` = the workers whose collected output carries ≥1 real finding (a `## Findings index` that isn't empty / `(none)` / a crash tail). If fewer than **2** producers, **skip**: append `[verify skipped — needs ≥2 finding-producing workers, had {k}]` to progress and return empty verdicts. A lone reviewer has no independent peer to check it; verifying a worker with its own model defeats the purpose.

2. **Assignment — deterministic, balanced, never self.** Index `producers` `0..M-1` in dispatch order. Each worker `j`'s findings are verified by exactly one peer: by default the next producer cyclically, `(j+1) mod M`. This single M-cycle gives full coverage, exactly one verification per worker, and guarantees no worker verifies its own findings. **Do not randomize** — a fixed cycle is reproducible and load-balanced.
   - **Model-diversity preference**: when the panel mixes models, prefer that each verifier check a *different* model than its own. Clean way: before applying the `+1` cycle, reorder `producers` so identical models aren't cyclically adjacent (interleave them), then cycle. If no clean interleave exists (e.g. three `claude` + one `gemini`), keep the plain `(j+1) mod M` cycle — **coverage and balance always win over diversity**; never drop or double-assign a worker to chase a different model.
   - **All-one-model panel** (e.g. `claude,claude`): the cyclic rule still applies — a fresh, independent instance with no shared context is a real adversarial check, just not cross-model. Note `[verify: single-model panel — cross-instance, not cross-model]` in progress.

3. **Dispatch** (batch-spawn then batch-send, exactly as SPAWN-WORKERS + DISPATCH-WAVE — issue all verifier `spawn_agent` calls together, then all `send_input` calls together; one pairing per assignment, all parallel). For each pairing (verifier agent → target worker): `spawn_agent(agent_tool_id={verifier.agent_tool_id}, name="counselor-verify-{verifier_agent}-{target_index}", include_agent_instructions=true)` → fresh `verifier_pid` + optional `agent_instructions`. Apply the `strict` `extra_args` allowlist gated on the verifier's `tool_type` exactly as for workers (verifiers are read-only too; a non-Claude verifier under `strict` downgrades to best-effort, never skipped). Build the first turn by substituting into `verifier_preamble_text`: `{{TARGET_LABEL}}` ← the target's `{agent}-{index}`; `{{TARGET_FINDINGS}}` ← that worker's collected output (prose + findings index). Prepend `agent_instructions` (then a blank line) if non-empty. `send_input(..., submit=true, wait_ms=2000)`; `get_process_status` to confirm alive + accepted; on failure append an error line to progress and **continue** (one failed verifier leaves its target's findings unverified — noted — but does not abort the wave). Track `{verifier_agent, target_index, verifier_pid}`.

4. **Collect** via the COLLECT-WAVE idle-drain (no `on_finish` hook — verifiers don't iterate) with `wave_budget_ms` as the ceiling: arm `timer_fire_when_idle_any`, harvest the `already_idle` finishers first, fence-extract, and one-shot-nudge any verifier idle with no closing fence. Then for each verifier: `scratchpad_write(name="counselors.{run_id}.verify.{verifier_agent}-{target_index}", content=extracted, tags=["counselor","{run_id}","verify","{verifier_agent}"])` → **record the `scratchpad_id`** in a `verify_ids` list; parse its `## Verdict index` and keep the rulings **in memory**, keyed by `(target worker, finding id)`; append `[verify {verifier_agent}→{target_agent}-{target_index} done | status={status} | confirmed={c} refuted={r} uncertain={u}]` to progress; `close_process(verifier_pid)`.

**Returns**: `verdicts` (in memory: per target worker, per finding id → `{ruling, severity_adjustment, reason}`; empty if skipped) + `verify_ids` (the verify-pad scratchpad ids, for archiving on a clean `done`).

## Routine: SYNTHESIZE

**Inputs**: the collected outputs you already hold (no re-read), `agents_csv`, `rounds`, `preset_note`, and `verdicts` (the per-finding rulings from VERIFY-WAVE — empty if verification was `off` or skipped).

Produce the summary following `references/synthesis-prompt.md`, substituting `{{AGENTS_CSV}}`, `{{N}}`, `{{ROUNDS}}`, `{{PRESET_NOTE}}`, `{{TASK_SHORT_DESCRIPTION}}`. When `verdicts` is non-empty, attach each finding's ruling (by id) and follow the synthesis prompt's **verification-aware** path: tag confirmed findings, segregate refuted ones into the "Disputed / likely false-positive" section, and apply any verifier severity adjustments. When `verdicts` is empty (verification off or skipped), produce the report without those verification tags. Then:

`scratchpad_write(name="counselors.{run_id}.summary", content=summary_md, tags=["counselor","{run_id}","summary"])` → record `summary_id`. **Never archive the summary** — it is the deliverable and stays visible.

## Routine: CLASSIFY-AND-FINALIZE

Settles the run's terminal status and archives **only** on a clean `done`. Both modes call this once, after SYNTHESIZE — the single source of truth for the done/partial/cancelled decision so the two callers can't drift. Each caller owns *detecting* the cut-short/cancel signals; this routine owns what they *mean*.

**Inputs**: `worker_outcomes` (every worker's final status across the whole run — all waves/rounds), `cut_short_status` (empty if the run reached its natural end; otherwise the label for a run stopped *as a whole* before collection finished — `timeout` when the budget was exhausted via the deadline timer, `partial` when loop `on_timeout == abort` cut the pipelined drain short at the run deadline (discarding in-flight partial rounds), or run-mode's wave guard expired with workers unfinished), `cancelled` (true if the user cancelled), `worker_ids` (all worker `scratchpad_id`s), `verify_ids` (all verify-pad `scratchpad_id`s from VERIFY-WAVE — empty if verification was off/skipped), `progress_id`, and — loop only — `prompt_id`.

1. `any_usable` = at least one worker produced a real extracted analysis (not an empty/crash tail).
2. Classify:
   - **`cancelled`** — if `cancelled`. (Synthesis already ran upstream on whatever was held.) **Do not archive.**
   - **`cut_short_status`** (`timeout` | `partial`) — else if `cut_short_status` is set: the run was stopped as a whole. **Do not archive** — leave the per-agent pads visible for inspection.
   - **`partial`** — else if not `any_usable`: collection finished but nobody produced usable output. **Do not archive.**
   - **`done`** — otherwise: collection finished and at least one worker produced usable output. Individual workers that crashed or timed out *among* successful peers do **not** downgrade the run — they're reported in the synthesis Panel-health section.
3. On **`done`** only: **ARCHIVE-WORKING-PADS** with `worker_ids` + `verify_ids` + `progress_id` (+ `prompt_id` in loop).
4. `kv_set("counselors.{run_id}.status", <classified status>)`.

**Returns**: the terminal status. (KV-key cleanup on a clean `done` is the *skill's* job — Step 9 — not this routine's.)

## Routine: ARCHIVE-WORKING-PADS  *(clean `done` path only)*

**Inputs**: all worker `scratchpad_id`s (every wave/round), `verify_ids` (all verify-pad ids — empty if verification didn't run), `progress_id`, and — loop only — `prompt_id`.

For each of those ids: `scratchpad_archive(scratchpad_id=...)`. This leaves `counselors.{run_id}.summary` as the **only visible** pad for the run. Archiving **hides, does not delete** — archived pads stay in Solo. Note that `scratchpad_list(tags=["{run_id}"])` *excludes* archived pads, so to re-open one later you read it by the `scratchpad_id` you recorded (or un-archive it in Solo).

**Only run this on a clean `done`.** For `partial` / `timeout` / `cancelled`, archive nothing — leave the per-agent pads visible for inspection.

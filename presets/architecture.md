# Preset: architecture

A review workflow for finding deepening opportunities — refactors that turn shallow modules into deep ones, for better testability and navigability.

## Discovery focus

When exploring the repo, hunt for architectural friction (go deep on the area the task names first; sweep the rest of this list broadly only if discovery budget remains). If a domain glossary (e.g. CONTEXT.md) or architecture decision records (e.g. docs/adr/*) exist, read them first to borrow the project's own nouns and to avoid re-suggesting refactors a decision already rejected:
- Shallow modules — the interface (everything a caller must know: types, invariants, error modes, ordering, config) is nearly as complex as the implementation behind it
- Concept fragmentation — understanding one idea forces bouncing across many tiny modules
- Behaviour split away from the place that owns its invariant or ordering — e.g. logic pulled into pure helpers for testability while the rule it enforces lives elsewhere, so no single module owns it (good pure functions are fine; the smell is the ownerless invariant)
- Tightly-coupled modules that leak across their seams — one module reaching into another's internals
- Areas that are untested, or hard to test through their current interface
- Single-adapter indirection masquerading as a seam (a port with only one implementation)

## What the written prompt should emphasize

The standard per-finding fields (including `reachability`) and the severity/confidence anchors come from the execution boilerplate — don't restate them. On top of those, instruct the panel to:
- Reason in and label findings with this architecture vocabulary: **module** (anything with an interface + implementation — function, class, package, or slice), **interface** (everything a caller must know, not just the signature), **depth** (much behaviour behind a small interface = deep; interface nearly as complex as implementation = shallow), **seam** (where an interface lives — a place to alter behaviour without editing in place), **adapter** (a concrete thing satisfying an interface at a seam), **leverage** (what callers gain from depth), **locality** (change, bugs, and knowledge concentrating in one place). Cite code by the repo's own names, but classify it with these terms so the panel stays comparable; keep the distinctions (a **seam** is not a DDD bounded context). Depth is **leverage** measured at the interface — behaviour reachable per unit a caller must learn — not a ratio of implementation lines to interface lines; don't reward a padded implementation. Map these terms onto the project's native shapes: a CLI's interface is its flags, config, env vars, exit codes, and file layout; a library's is its public API plus documented invariants and error behaviour; a data pipeline's is its stage contracts, schemas, connectors, and retry/idempotency — don't force service-app "ports" onto a script or a pipeline.
- Apply the **deletion test** as the core evidence for every candidate: imagine deleting the module. If complexity vanishes, it was a pass-through — don't deepen it. If complexity reappears across N callers, it earns its keep — name those N call sites. For `reachability`, point to where the complexity is actually encountered today — call sites, command paths, pipeline stages, tests, config paths, or recurring change scenarios; the N call sites are evidence of the locality cost, not the whole story. (This is also the refutation step: a "shallow module" finding survives only if the deletion test shows the complexity really would scatter.)
- For each candidate report: the files/modules involved; the **problem** (why it's shallow, or where it leaks); the **deepening** (what merges behind one interface); the **wins** in terms of locality and leverage, quantified where possible — call sites unified (N), shallow wrappers/modules deleted (M), test suites collapsed onto one interface; and a **recommendation strength** tag — `Strong`, `Worth exploring`, or `Speculative` — calibrated against execution risk, not just payoff: a low-risk in-process merge with clear leverage is `Strong`; a deepening that needs a new cross-network port is rarely more than `Worth exploring`. Rank the deepening candidates by recommendation strength and leverage (tie-broken by execution risk), not by severity — severity is secondary for a structural finding. Then give an execution **sequence** across them, not a flat list: the first safe move, what it unlocks, what should wait, the migration order, and where code gets deleted.
- Rate **severity** by the harm the current structure already causes or risks — not the refactor's upside (recommendation strength carries that). Most deepening findings are low/medium; reserve high for concrete present-day pain: recurring bugs traced to the structure, unsafe edits on a common path, or a testability gap that lets real regressions ship.
- Report a correctness bug only when you surface it while tracing the structure — a duplicated invariant, split ownership, a leaky seam — and it is evidence that the missing module matters; list such bugs **separately** from the deepening candidates and rank those by severity. Don't broaden into general defect hunting — that's the `bughunt` preset.
- Classify each candidate's dependencies and state the resulting testing strategy: in-process (merge and test through the new interface directly), local-substitutable (test with a local stand-in), remote-but-owned (define a port; a real adapter in prod, an in-memory adapter in tests), or true-external (inject a port; a mock adapter in tests). The interface is the test surface — new tests at the deepened interface replace the old shallow-module unit tests. Tests assert observable outcomes through the interface and should survive internal refactors — if a test must change when the implementation changes, it was testing past the interface.
- Respect seam discipline: a port is justified only when something genuinely varies across the seam — the dependency is remote, slow, nondeterministic, stateful, or owned elsewhere. Two adapters (prod + test) are the usual sign, but wanting to mock something in a test is not, by itself, enough — one adapter is indirection, not a seam. And don't widen a module's interface to expose its internals just so tests can reach them — those are internal seams, kept private to the implementation.
- If a candidate contradicts an architecture decision record found during discovery, flag it explicitly and surface it only when the friction is real enough to warrant reopening that decision — don't enumerate every refactor a record forbids.
- Make each **fix** a smallest shippable refactor, not a direction: name the target module/interface, the first callers to migrate, what gets deleted, and the tests to add or move to the new interface — plus one explicit non-goal. If you can't state that concretely from the code, mark the candidate `Speculative` or drop it.
- Skip cosmetic restructuring with no locality or leverage payoff.
